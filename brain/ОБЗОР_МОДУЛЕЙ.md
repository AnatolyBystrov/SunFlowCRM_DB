**Высокоуровневое описание архитектуры и взаимодействия модулей**

## 🎯 Общая Архитектура

Платформа Sun UW построена на модульной архитектуре из **13 основных модулей**, которые взаимодействуют друг с другом для обеспечения полного цикла работы с перестрахованием - от первого контакта с клиентом до выплаты убытков.

```
┌─────────────────────────────────────────────────────────────┐
│                    Sun UW Platform                          │
│                                                             │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐  │
│  │   CRM    │→│Underwriting│→│ Financial│→│  Audit   │  │
│  └──────────┘  └──────────┘  └──────────┘  └──────────┘  │
│       ↓              ↓              ↓                       │
│  ┌──────────────────────────────────────────────────┐     │
│  │         Reference Data & Compliance              │     │
│  └──────────────────────────────────────────────────┘     │
└─────────────────────────────────────────────────────────────┘
```

## 📦 Модули и Их Назначение

### 1️⃣ Базовая Инфраструктура (Core Infrastructure)

**Таблицы:** `tenants`, `users`, `teams`, `team_members`

**Назначение:**
Фундамент платформы - управление организациями (тенантами), пользователями и командами. Обеспечивает мультитенантность и разграничение доступа.

**Ключевые функции:**
- ✅ Изоляция данных между организациями
- ✅ Управление пользователями (синхронизация с Clerk)
- ✅ Ролевая модель (Admin, Manager, Underwriter, Sales, Member)
- ✅ Организация команд внутри компании

**Взаимодействие:**
- **→ Все модули:** Каждая запись в системе привязана к `tenantId`
- **→ CRM:** Пользователи владеют сделками (`Deal.ownerId`)
- **→ Underwriting:** Пользователи назначаются андеррайтерами (`Submission.underwriterId`)
- **→ Audit:** Все операции логируются с `performedBy` (userId)

**Пример:**
```
Tenant "Reinsurance Corp"
  ├─ User "John (Admin)"
  ├─ User "Sarah (Underwriter)"
  └─ Team "Underwriting Team"
      ├─ Sarah (Leader)
      └─ Mike (Member)
```

---

### 2️⃣ CRM - Управление Продажами (Sales CRM)

**Таблицы:** `pipelines`, `stages`, `deals`, `persons`, `organizations`, `insurance_products`, `deal_coverages`

**Назначение:**
Воронка продаж в стиле Pipedrive для управления потенциальными сделками. Первая точка контакта с клиентом.

**Ключевые функции:**
- ✅ Kanban-доска сделок по этапам
- ✅ Управление контактами (Person) и компаниями (Organization)
- ✅ Каталог страховых продуктов
- ✅ Расчет покрытий и премий
- ✅ Конвертация выигранных сделок в заявки на андеррайтинг

**Взаимодействие:**
- **← Core:** Сделки принадлежат тенанту и владельцу
- **→ Underwriting:** При статусе `WON` создается `Submission`
- **→ Financial:** Можно выставить счет напрямую с Deal
- **→ Reference Data:** Использует `LineOfBusiness`, `ClassOfBusiness`
- **→ Activity:** Звонки, встречи, задачи привязаны к Deal
- **→ Compliance:** Проверка Organization/Person перед сделкой

**Workflow:**
```
Lead → Qualified → Proposal → Negotiation → WON
                                              ↓
                                        Submission
```

**Пример:**
```
Pipeline "Reinsurance Sales"
  ├─ Stage "Lead" (10%)
  │   └─ Deal "Property Insurance - ABC Corp" ($500K)
  ├─ Stage "Proposal" (50%)
  │   └─ Deal "Marine Coverage - XYZ Ltd" ($1M)
  └─ Stage "Won" (100%)
      └─ Deal "Casualty - DEF Inc" ($750K) → создает Submission
```

### 3️⃣ Андеррайтинг (Underwriting)

**Таблицы:** `submissions`, `policies`, `claims`

**Назначение:**
Оценка рисков, выпуск полисов и обработка убытков. Ядро страхового бизнеса.

**Ключевые функции:**
- ✅ Оценка рисков (Risk Level: Low → Critical)
- ✅ Расчет премий и комиссий
- ✅ Выпуск полисов с финансовым трекингом
- ✅ Обработка претензий/убытков

**Взаимодействие:**
- **← CRM:** Получает Submission от выигранных Deal
- **→ Financial:** Policy генерирует Invoice для премий
- **→ Financial:** Claim генерирует Invoice для выплат
- **→ Documents:** Прикрепление документов к Submission/Claim
- **→ Activity:** Задачи андеррайтера
- **→ Reference Data:** Классификация по Line/Class of Business
- **→ Compliance:** Проверка рисков влияет на `riskLevel`

**Workflow:**
```
Submission (NEW)
    ↓ (Underwriter reviews)
Submission (REVIEW)
    ↓ (Risk analysis)
Submission (ANALYSIS)
    ↓ (Underwriting decision)
Submission (APPROVED)
    ↓
Policy (ACTIVE)
    ↓ (If loss occurs)
Claim (REPORTED → INVESTIGATING → SETTLED)
```

**Пример:**
```
Submission #SUB-001
  ├─ Company: "ABC Corp"
  ├─ Amount: $500,000
  ├─ Risk Level: MEDIUM
  ├─ Stage: APPROVED
  └─ Creates → Policy #POL-001
                  ├─ Premium: $12,500
                  ├─ Status: ACTIVE
                  └─ If loss → Claim #CLM-001
```

### 4️⃣ Финансовый Модуль (Financial Module)

**Таблицы:** `invoices`, `invoice_items`, `payments`, `scheduled_payments`, `credit_notes`, `payment_methods`

**Назначение:**
Управление всеми финансовыми операциями - счета, платежи, расчеты, мультивалютность.

**Ключевые функции:**
- ✅ Выставление счетов (разовых и рекуррентных)
- ✅ Прием платежей с конвертацией валют
- ✅ График платежей (installments)
- ✅ Кредит-ноты (возвраты, корректировки)
- ✅ Мультивалютность с FX rates

**Взаимодействие:**
- **← Underwriting:** Policy/Claim генерируют Invoice
- **← CRM:** Deal может генерировать Invoice
- **→ Audit Trail:** Все операции логируются
- **→ Reference Data:** Использует Currency, ExchangeRate
- **← Core:** Платежи создаются пользователями

**Workflow:**
```
Policy/Deal
    ↓
Invoice (DRAFT)
    ├─ InvoiceItem 1: Premium
    ├─ InvoiceItem 2: Fee
    └─ Total: $12,500
    ↓ (Send to client)
Invoice (SENT)
    ↓ (Client pays)
Payment
    ├─ Amount: €11,500
    ├─ FX Rate: 0.92
    ├─ USD Equivalent: $12,500
    └─ Logs → PaymentAuditLog
    ↓
Invoice (PAID)
```

**Пример рекуррентного счета:**
```
Invoice #INV-001
  ├─ Recurring: Yes
  ├─ Repeat: Every 30 DAYS
  ├─ Cycles: 12
  ├─ Next Date: 2026-03-06
  └─ Auto-generates → INV-002, INV-003, ...
```

### 5️⃣ Справочные Данные (Reference Data)

**Таблицы:** `currencies`, `exchange_rates`, `lines_of_business`, `classes_of_business`, `regions`, `countries`

**Назначение:**
Глобальные и тенант-специфичные справочники для классификации и конвертации.

**Ключевые функции:**
- ✅ Валюты и курсы обмена
- ✅ Классификация страхования (Line/Class of Business)
- ✅ Географическая классификация (Region/Country)
- ✅ Поддержка как глобальных, так и кастомных справочников

**Взаимодействие:**
- **→ Financial:** Currency, ExchangeRate для платежей
- **→ CRM:** LineOfBusiness, ClassOfBusiness для Deal
- **→ Underwriting:** Классификация Submission
- **→ Все модули:** Country, Region для адресов

**Структура:**
```
LineOfBusiness "Property"
  ├─ ClassOfBusiness "Fire"
  ├─ ClassOfBusiness "Earthquake"
  └─ ClassOfBusiness "Flood"

ExchangeRate
  ├─ 2026-02-06: USD → EUR = 0.92
  ├─ 2026-02-06: USD → GBP = 0.79
  └─ 2026-02-06: USD → AED = 3.67
```

### 6️⃣ Комплаенс и KYC (Compliance & KYC)

**Таблица:** `compliance_checks`

**Назначение:**
Проверки на соответствие регуляторным требованиям для клиентов.

**Ключевые функции:**
- ✅ Проверка PEP (Politically Exposed Person)
- ✅ Проверка санкций
- ✅ Проверка налогового резидентства
- ✅ Верификация лицензий
- ✅ Определение конечного бенефициара (UBO)

**Взаимодействие:**
- **← CRM:** Проверка Organization/Person
- **→ Underwriting:** Влияет на `Submission.riskLevel`
- **← Core:** Проверки выполняются пользователями

**Workflow:**
```
Organization "ABC Corp"
    ↓
ComplianceCheck
    ├─ Status: IN_REVIEW
    ├─ isPEP: false
    ├─ hasSanctions: false
    ├─ License: Valid until 2027
    └─ Risk Level: LOW
    ↓
Status: APPROVED
    ↓
Can proceed with Deal/Submission
```

### 7️⃣ Аудит Операций (Audit Trail)

**Таблицы:** `payment_audit_logs`, `invoice_audit_logs`, `credit_note_audit_logs`

**Назначение:**
Полное логирование всех финансовых операций для комплаенс и безопасности.

**Ключевые функции:**
- ✅ Логирование всех изменений (CRUD)
- ✅ Хранение diff (до/после)
- ✅ Запись IP-адреса и User Agent
- ✅ Неизменяемый audit trail

**Взаимодействие:**
- **← Financial:** Автоматическое логирование Payment/Invoice/CreditNote
- **← Core:** Запись `performedBy` (userId)

**Пример:**
```
PaymentAuditLog
  ├─ Payment #PAY-001
  ├─ Action: UPDATED
  ├─ Changes:
  │   ├─ Before: { status: "PENDING" }
  │   └─ After: { status: "COMPLETED" }
  ├─ Performed By: user_123
  ├─ Performed At: 2026-02-06 14:00:00
  └─ IP: 192.168.1.1
```

**Таблицы:** `documents`, `document_versions`

**Назначение:**
Управление документами с версионированием.

**Ключевые функции:**
- ✅ Загрузка файлов (PDF, Word, Excel, Images)
- ✅ Версионирование документов
- ✅ Извлечение текста (OCR/parsing)
- ✅ Привязка к Submission/Claim

**Взаимодействие:**
- **← Underwriting:** Документы к Submission/Claim
- **← Core:** Загрузка пользователями

**Пример:**
```
Document "Policy Agreement.pdf"
  ├─ Version 1 (2026-01-15)
  ├─ Version 2 (2026-02-01) - amendments
  └─ Version 3 (2026-02-06) - final
  └─ Attached to: Submission #SUB-001
```

CRM-активности, заметки, файлы и комментарии для коллаборации.

**Ключевые функции:**
- ✅ Планирование звонков, встреч, задач
- ✅ Заметки к сделкам/заявкам
- ✅ Прикрепление файлов
- ✅ Комментарии к заявкам

**Взаимодействие:**
- **← CRM:** Активности к Deal/Person/Organization
- **← Underwriting:** Активности к Submission/Claim
- **← Core:** Создаются пользователями

**Пример:**
```
Deal "ABC Corp - Property"
```

### 1️⃣1️⃣ Рабочие Процессы (Workflows)

**Таблицы:** `workflows`, `workflow_executions`

**Назначение:**
Интеграция с n8n для автоматизации бизнес-процессов.

**Ключевые функции:**
- ✅ Автоматические уведомления
- ✅ Интеграция с внешними системами
- ✅ Автоматизация рутинных задач
- ✅ Логирование выполнения

**Взаимодействие:**
- **→ Все модули:** Может триггериться на события в любом модуле
- **← Core:** Workflows принадлежат тенанту

**Примеры workflow:**
```
Workflow "New Submission Notification"
  ├─ Trigger: Submission.stage = "NEW"
  ├─ Action: Send email to underwriter
  └─ Execution: SUCCESS

Workflow "Invoice Overdue Reminder"
  ├─ Trigger: Invoice.dueDate < today AND status != "PAID"
  ├─ Action: Send reminder email
  └─ Execution: SUCCESS
```

## 🔄 Полный Цикл Взаимодействия Модулей

### Сценарий: От Лида до Выплаты Убытка

```
1. CRM - Первый Контакт
   ├─ Person "John Doe" создан
   ├─ Organization "ABC Corp" создана
   └─ ComplianceCheck → APPROVED

2. CRM - Продажа
   ├─ Deal создана в Pipeline
   ├─ Deal проходит Stages
   ├─ Activities: Calls, Meetings
   └─ Deal.status = WON

3. Underwriting - Оценка
   ├─ Submission создана из Deal
   ├─ Documents прикреплены
   ├─ Underwriter reviews
   ├─ RiskLevel определен
   └─ Submission.stage = APPROVED

4. Underwriting - Выпуск Полиса
   ├─ Policy создан
   └─ Policy.status = ACTIVE

5. Financial - Выставление Счета
   ├─ Invoice создан для премии
   ├─ InvoiceItems добавлены
   └─ Invoice.status = SENT

6. Financial - Оплата
   ├─ Payment получен (EUR)
   ├─ ExchangeRate применен (EUR→USD)
   ├─ PaymentAuditLog записан
   └─ Invoice.status = PAID

7. Underwriting - Убыток
   ├─ Claim создан
   ├─ Documents прикреплены
   ├─ Activities для обработки
   └─ Claim.status = APPROVED

8. Financial - Выплата
   ├─ Invoice создан для выплаты
   ├─ Payment или CreditNote
   ├─ PaymentAuditLog записан
   └─ Claim.status = SETTLED

9. Support - Обратная Связь
   └─ Ticket создан для follow-up
```

## 🎯 Ключевые Принципы Архитектуры

### 1. Multi-Tenancy
Все модули (кроме глобальных справочников) изолированы по `tenantId`

### 2. Audit Trail
Все финансовые операции логируются автоматически

### 3. Soft Delete
Данные не удаляются физически, используется флаг `deleted`

### 4. Custom Fields
Любая сущность может быть расширена через JSONB поля

### 5. Workflow Integration
Любое событие может триггерить автоматизацию через n8n
