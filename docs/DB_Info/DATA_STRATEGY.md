# Sun UW Platform - Data Architecture & Reporting Strategy

Sun UW использует архитектуру данных, оптимизированную для multi-tenant B2B SaaS с высокой отчётностью и строгими требованиями к безопасности.

---

## 1. Разделение контуров: Auth vs Domain

### Auth & Identity (SuperTokens DB)
- Все данные аутентификации (учётные записи, сессии, токены, рецепты) хранятся в собственной БД SuperTokens и не попадают в доменную базу.
- В доменной БД Sun UW хранится только проекция пользователя: `User` с полями `id`, `supertokensUserId`, `tenantId`, ролевой моделью (RBAC) и доменными атрибутами (имя, настройки, статусы).
- Такое разделение упрощает безопасность, ротацию ключей и независимое масштабирование auth-слоя.

### Core Domain Database (PostgreSQL 16)
- Единая доменная БД для всех модулей: Clients, KYC, Underwriting, Claims, Documents, Accounting, Approvals, Tasks.
- Multi-tenant реализован по паттерну **shared database / shared schema + tenantId**: практически все бизнес-таблицы содержат `tenantId` и индексируются по нему.
- Функциональная логика (NestJS) и при необходимости RLS в PostgreSQL гарантируют, что каждый запрос выполняется в контексте конкретного tenant’а.

---

## 2. Multi-Tenancy Strategy

Основной паттерн — **shared schema, tenant-scoped rows**:
- `tenantId` является обязательным полем для ключевых сущностей (clients, submissions, claims, documents, invoices и т.д.).
- Для уникальных полей используется **tenant-scoped уникальность**, например: `UNIQUE (tenantId, email)` для пользователей внутри организации или `UNIQUE (tenantId, invoiceNumber)` для учёта инвойсов.
- В перспективе дополнительно возможен слой **Row-Level Security (RLS)** по `tenantId`, чтобы безопасно подключать внешние BI-инструменты и обеспечивать жёсткую изоляцию на уровне БД.

## 3. OLTP vs Reporting (CQRS Light)

### Операционный контур (OLTP)
- Основная Postgres-БД обслуживает транзакционные сценарии: CRM, андеррайтинг, урегулирование убытков, документооборот, биллинг.
- Запросы оптимизируются под типовые фильтры и показатели (по `tenantId`, статусам, стадиям pipeline, датам).

### Отчётность и аналитика (CQRS Light / OLAP-слой)
- На первом этапе отчёты реализуются через материализованные представления и отдельную схему внутри той же БД: агрегаты по портфелям, loss-ratio, производительность pipeline и т.д.
- При росте нагрузки предусмотрен переход к выделенному reporting-слою:
  - Чтение с реплики основной БД для тяжёлых аналитических запросов;
  - Либо отдельная аналитическая БД/хранилище, куда данные попадают через ETL/CDC.

Такой подход изолирует тяжёлую аналитику от критичных OLTP-операций и позволяет масштабировать отчётность независимо от основной нагрузки.

---

## 4. Масштабирование и эволюция

По мере роста возможны эволюции без ломки модели:
- Вынос отдельных доменов (например, Accounting или Claims) в собственные сервисы/БД по паттерну **“database per service”**, сохраняя текущую схему как стартовую точку;
- Добавление специализированного аналитического хранилища (DWH) для кросс-tenant отчётов и продвинутой аналитики.

**Базовый дизайн (shared schema + tenantId + отделённый Auth) выбран так, чтобы:**
1. Поддерживать многоклиентный режим без дублирования схем;
2. Упрощать построение отчётов по tenant’ам и в разрезе портфелей;
3. Позволять постепенно переходить от “CQRS Light” к полноценному разделению OLTP/OLAP, когда это станет оправдано по нагрузке.
